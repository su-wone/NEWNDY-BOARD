## lint, type-check, build — 코드 검사 3단계

### lint (ESLint)
코드의 **나쁜 습관과 잠재적 버그**를 찾아내는 검사.

```tsx
const name = "hello"   // ← 선언만 하고 안 쓰면: "미사용 변수입니다"
console.log(nam)       // ← 오타: "nam은 정의되지 않았습니다"
<img src="photo.jpg" />  // ← "alt 속성이 없습니다" (접근성 문제)
```

### type-check (TypeScript)
**데이터 타입이 맞는지** 검사.

```tsx
function add(a: number, b: number) { return a + b }
add("hello", 5)  // ← 에러: string을 넣으면 안 됨, number만 가능
```

### build
개발용 코드(TypeScript, JSX)를 브라우저가 이해할 수 있는 **배포용 코드(JS, HTML, CSS)로 변환**. build가 실패하면 = 배포할 수 없는 코드.

### 왜 이 순서인가?
```
lint → type-check → build
(가벼움)  (보통)    (무거움)
```
가벼운 검사부터 실행. lint에서 실패하면 무거운 build까지 갈 필요 없음 → 시간 절약.

## ESLint vs Prettier

| | ESLint | Prettier |
|---|---|---|
| 역할 | 코드 **품질** 검사 | 코드 **모양** 정리 |
| 예시 | 미사용 변수, React Hook 규칙 위반 | 따옴표, 세미콜론, 들여쓰기 |
| 비유 | 글의 **문법/논리 오류** | 글의 **맞춤법/띄어쓰기** |

## Husky + lint-staged — 자동화 파이프라인

```
git commit 실행
  → Husky가 pre-commit hook 감지 (트리거)
    → "npx lint-staged" 실행
      → lint-staged가 커밋할 파일만 골라서 ESLint 실행 (자동화 라인)
        → 통과하면 커밋 완료
        → 실패하면 커밋 차단
```

- **Husky** = "커밋할 때 뭔가를 실행해라"라는 **트리거**
- **lint-staged** = "커밋할 파일에 어떤 검사를 돌려라"라는 **자동화 라인**
- **ESLint** = 실제로 코드를 검사하는 **도구**

## GitHub Actions CI

GitHub에 코드를 push하거나 PR을 만들면, **자동으로 코드를 검사하고 빌드하는** 서비스. 별도 서버 없이 GitHub이 제공하는 가상 환경에서 실행.

### CI(Continuous Integration)
"지속적 통합" — 코드를 합치기(merge) 전에 자동으로 검사해서 문제를 미리 발견.

## Vercel — 배포

### 프로덕션 배포 vs Preview 배포

| | 프로덕션 배포 | Preview 배포 |
|---|---|---|
| 트리거 | main에 머지 | PR 생성 |
| URL | 고정 URL | 임시 URL (PR마다 다름) |
| 용도 | 실제 사용자가 접속하는 서비스 | 코드 리뷰 시 화면 미리보기 |

- **main 머지** → 진짜 서비스 업데이트 (프로덕션 배포)
- **PR 생성** → 미리보기 사이트 생성 (Preview 배포)

## .env.example

`.env` 파일은 API 키, 비밀번호 같은 비밀 정보를 담아서 Git에 올리지 않음. `.env.example`은 **변수 목록만 보여주는 템플릿**. 새 팀원은 이 파일을 복사해서 `.env.local`로 만들고 자기 값을 채움.
