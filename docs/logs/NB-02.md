## 교훈: AI가 제안하는 대로 따라가지 말 것

create-next-app이 처음에 "Next.js 14"를 설치하려 했다. AI가 최신 버전을 정확히 알지 못할 수 있으므로, **각 기술의 현재 버전과 사용법을 직접 확인하는 습관**이 중요하다.

실제로 확인해보니:
- AI가 제안한 Next.js 14 → 실제 최신은 **Next.js 16.1.7**
- AI가 알려준 Node.js LTS 22 → 실제 최신 LTS는 **Node.js 24.14.0**

> **주도적으로 버전을 확인하고, 왜 그 버전을 쓰는지 이해한 뒤 진행하자.**

---

## Node.js 업그레이드 (v20 → v24)

기존 v20.20.0이 설치되어 있었는데, 최신 LTS인 v24.14.0으로 업그레이드했다.

```bash
nvm install 24
```
> nvm(Node Version Manager)으로 v24를 다운로드하고 전환한다.

```bash
nvm alias default 24
```
> 새 터미널을 열어도 항상 v24를 사용하도록 기본값으로 설정한다.

Node.js 버전을 바꾸면 **corepack도 다시 활성화**해야 한다. corepack은 Node.js 버전마다 따로 관리되기 때문이다.

```bash
corepack enable
```

## 핵심 개념 정리

### npx란?
`npx`는 패키지를 **설치하지 않고 바로 실행**하는 명령어다. `create-next-app`은 프로젝트 생성 시 한 번만 쓰면 되므로, 설치 없이 실행하는 게 적합하다.

### App Router란?
Next.js에는 두 가지 라우팅 방식이 있다:
- **Pages Router** (구버전) — `pages/` 폴더 기반. 파일 이름 = URL 경로
- **App Router** (신버전) — `app/` 폴더 기반. 레이아웃 중첩, 서버 컴포넌트 등 최신 기능 지원

App Router를 쓰면 레이아웃(사이드바 등)을 `layout.tsx`로 한 번 정의하고, 모든 하위 페이지에 자동으로 적용할 수 있다. 이것이 NB-05(사이드바 라우팅)에서 핵심이 된다.

### TypeScript strict: true란?
TypeScript의 타입 검사를 가장 엄격하게 설정하는 옵션이다. `null` 체크, 암묵적 `any` 금지 등을 강제해서 런타임 에러를 사전에 방지한다. 처음엔 까다롭지만 프로젝트가 커질수록 버그를 줄여준다.

## Next.js 프로젝트 생성

```bash
npx create-next-app@latest apps/web --typescript --tailwind --eslint --app --src-dir --import-alias "@/*" --use-pnpm
```

각 옵션의 의미:
| 옵션 | 의미 |
|---|---|
| `--typescript` | TypeScript 사용 |
| `--tailwind` | Tailwind CSS 포함 |
| `--eslint` | ESLint(코드 검사 도구) 포함 |
| `--app` | App Router 사용 (Next.js 최신 라우팅 방식) |
| `--src-dir` | 코드를 src 폴더 안에 배치 |
| `--import-alias "@/*"` | import 경로를 `@/`로 시작하도록 단축 |
| `--use-pnpm` | pnpm 패키지 매니저 사용 |

### 설치 중 발생한 문제

1. **React Compiler 사용 여부 프롬프트** — 아직 실험적 기능이라 No 선택
2. **pnpm ENOENT 에러** — Node.js를 v24로 바꾸면서 pnpm 연결이 끊어짐. `corepack enable`로 해결
3. **pnpm-workspace.yaml 중복 생성** — create-next-app이 web 안에도 워크스페이스 파일을 만들었는데, 루트에만 있으면 되므로 삭제

## 의존성 설치

```bash
pnpm install
```
> 모노레포에서는 루트에서 실행하면 모든 하위 프로젝트(web, types)의 의존성을 한번에 설치한다.

### 빌드 스크립트 승인 문제

설치 후 `sharp`(이미지 최적화)와 `unrs-resolver`의 빌드 스크립트를 승인해야 한다는 경고가 나왔다. 대화형 `pnpm approve-builds`가 작동하지 않아서, package.json에 직접 설정을 추가했다:

```json
"pnpm": {
  "onlyBuiltDependencies": ["sharp", "unrs-resolver"]
}
```
> 이렇게 하면 이 두 패키지의 빌드 스크립트 실행을 명시적으로 허용한다.

