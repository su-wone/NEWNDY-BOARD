> 작업일: 2026-03-18 | 상태: 완료

## 핵심 개념 정리

### ESLint vs Prettier — 역할이 다르다

| | ESLint | Prettier |
|---|---|---|
| 역할 | **코드 품질** 검사 | **코드 모양** 정리 |
| 예시 | 미사용 변수, 잘못된 import, React Hook 규칙 위반 | 따옴표, 세미콜론, 들여쓰기 |
| 비유 | 글의 **문법/논리 오류** | 글의 **맞춤법/띄어쓰기** |

ESLint는 코드의 **버그와 품질 문제**를 잡고, Prettier는 코드의 **시각적 통일성**을 맞춘다.

### Husky + lint-staged — 자동화 파이프라인

```
git commit 실행
  → Husky가 pre-commit hook 감지 (트리거)
    → "npx lint-staged" 실행
      → lint-staged가 커밋할 파일만 골라서 ESLint 실행 (자동화 라인)
        → 통과하면 커밋 완료
        → 실패하면 커밋 차단
```

- **Husky** = "커밋할 때 뭔가를 실행해라"라는 **트리거**
- **lint-staged** = "커밋할 파일에 어떤 검사를 돌려라"라는 **자동화 라인**
- **ESLint** = 실제로 코드를 검사하는 **도구**

### eslint-config-prettier — 충돌 방지

ESLint와 Prettier가 동시에 존재하면, 둘 다 "따옴표를 어떻게 쓸지" 같은 규칙을 가지고 있어서 충돌할 수 있다. `eslint-config-prettier`는 ESLint에서 Prettier와 겹치는 규칙을 꺼준다. ESLint 설정의 **맨 마지막**에 넣어야 앞의 모든 규칙을 덮어쓸 수 있다.

### .env.example — 팀원 온보딩용 템플릿

`.env` 파일은 API 키, 데이터베이스 주소 같은 비밀 정보를 담아서 Git에 올리지 않는다. 새 팀원은 "어떤 환경 변수가 필요한지" 알 수가 없으므로, `.env.example`에 변수 목록만 보여주는 템플릿을 만들어둔다. 팀원은 이 파일을 복사해서 `.env.local`로 만들고 자기 값을 채우면 된다.

## 교훈: ESLint v10 호환성 문제

처음에 ESLint v10을 설치하려 했지만, Next.js의 ESLint 플러그인들(`eslint-plugin-react`, `eslint-plugin-react-hooks`, `eslint-plugin-jsx-a11y` 등)이 아직 v10을 지원하지 않아서 peer dependency 에러가 발생했다.

> **최신 버전이 항상 좋은 것은 아니다. 프로젝트의 다른 도구들과 호환되는 버전을 선택해야 한다.**

결국 ESLint v9로 유지하기로 결정했다.

## ESLint 설정

`apps/web/eslint.config.mjs`에 이미 ESLint v9 (flat config 방식)이 설정되어 있었다. 여기에 `eslint-config-prettier`만 추가했다.

```js
import prettierConfig from "eslint-config-prettier";

const eslintConfig = defineConfig([
  ...nextVitals,        // Next.js 성능/접근성 규칙
  ...nextTs,            // TypeScript 규칙
  globalIgnores([...]), // .next, out, build 폴더 제외
  prettierConfig,       // Prettier 충돌 규칙 비활성화 (반드시 마지막)
]);
```

## Prettier 설치

```bash
npm install -D prettier eslint-config-prettier
```
> - `prettier` — 코드 포맷터
> - `eslint-config-prettier` — ESLint와 Prettier 충돌 방지
> - `-D` — 개발용으로 설치

`.prettierrc` 파일을 만들어 포맷 규칙을 정의했다:

```json
{
  "semi": false,
  "singleQuote": true,
  "tabWidth": 2,
  "trailingComma": "all",
  "printWidth": 80
}
```

| 옵션 | 의미 |
|---|---|
| `semi: false` | 세미콜론 안 붙임 |
| `singleQuote: true` | 작은따옴표 사용 |
| `tabWidth: 2` | 들여쓰기 2칸 |
| `trailingComma: "all"` | 마지막 항목 뒤에 쉼표 |
| `printWidth: 80` | 한 줄 최대 80자 |

### Prettier 활성화 상태

처음에는 shadcn/ui 코드와의 차이 때문에 lint-staged에서 Prettier를 빼두었다가, 이후 다시 활성화했다. 현재 커밋 시 `prettier --write` → `eslint --fix` 순서로 자동 실행된다.

## Husky + lint-staged 설치

```bash
npm install -D husky lint-staged
```
> - `husky` — Git hook 관리 도구
> - `lint-staged` — 커밋할 파일에만 검사를 실행하는 도구

### Husky 초기화

```bash
npx husky init
```
> `.husky/` 폴더를 만들고, `pre-commit` hook 파일을 생성한다. 또한 `package.json`에 `"prepare": "husky"` 스크립트를 자동 추가한다. 이 스크립트 덕분에 `npm install` 할 때 Husky가 자동으로 설정된다.

### pre-commit hook 설정

`.husky/pre-commit` 파일의 내용을 `npm test`에서 `npx lint-staged`로 변경했다.

### lint-staged 설정

`package.json`에 lint-staged 설정을 추가했다:

```json
"lint-staged": {
  "*.{ts,tsx}": ["eslint --fix"]
}
```

TypeScript 파일을 커밋할 때 ESLint가 자동으로 실행되고, `--fix` 옵션으로 자동 수정 가능한 문제는 수정해준다.

## .env.example 생성

`apps/web/.env.example` 파일을 만들었다:

- `NEXT_PUBLIC_APP_URL` — 웹 앱 주소 (`NEXT_PUBLIC_` 접두사 = 브라우저에서 접근 가능)
- `NEXT_PUBLIC_API_URL` — 백엔드 API 서버 주소
- `AUTH_*` — 인증 관련 (Sprint 2에서 설정 예정, 주석 처리)

## 티켓 요약
