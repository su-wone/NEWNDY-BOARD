## 핵심 개념 정리

### Vercel이란?

Next.js를 만든 회사에서 제공하는 **호스팅/배포 서비스**다. GitHub 저장소를 연결하면 코드를 push할 때마다 자동으로 빌드하고 인터넷에 공개해준다. 별도의 서버를 준비하거나 배포 스크립트를 작성할 필요가 없다.

### 프로덕션 배포 vs Preview 배포

| | 프로덕션 배포 | Preview 배포 |
|---|---|---|
| 트리거 | main에 머지(합침) | PR 생성 |
| URL | 고정 URL (예: newndy-board-web.vercel.app) | 임시 URL (PR마다 다름) |
| 용도 | 실제 사용자가 접속하는 서비스 | 코드 리뷰 시 화면 미리보기 |

**프로덕션** = 실제 사용자가 쓰는 진짜 서비스
**Preview** = 아직 main에 합치지 않은 코드를 미리 볼 수 있는 임시 사이트

### 왜 Preview 배포가 중요한가?

코드 리뷰할 때 코드만 보는 게 아니라, **실제 화면이 어떻게 바뀌는지** 직접 확인할 수 있다. Vercel이 PR에 Preview URL을 자동으로 코멘트로 달아주므로, 리뷰어가 클릭 한 번으로 결과를 볼 수 있다.

## 설정 과정

### 사전 조건

NB-02 ~ NB-05의 변경사항을 GitHub에 push해야 Vercel이 코드를 볼 수 있다. 커밋 내역:

```
[NB-02] Next.js 16 App Router 프로젝트 생성
[NB-03] Tailwind CSS + shadcn/ui 초기 설치 및 디자인 토큰 정의
[NB-04] ESLint + Prettier + Husky 코드 품질 도구 설정
[NB-05] GitHub Actions CI 파이프라인 구성
```

### 커밋 중 발생한 문제들

1. **apps/web/.git 중복 저장소** — `create-next-app`이 자동으로 git init을 해서, 모노레포 안에 git 저장소가 중복됨. `rm -rf apps/web/.git`으로 삭제

2. **lint-staged가 ESLint를 못 찾음** — 세 가지 원인이 겹침:
   - ESLint가 `apps/web`에만 설치되어 있고 루트에는 없었음 → 루트에도 설치
   - ESLint 설정 파일이 `apps/web`에만 있음 → lint-staged 설정을 `apps/web/package.json`으로 이동
   - pre-commit hook이 루트에서 실행됨 → `cd apps/web && npx lint-staged`로 변경

3. **.env.example이 .gitignore에 걸림** — `.env*` 패턴이 `.env.example`까지 무시. `!.env.example` 예외 규칙 추가

### Vercel 프로젝트 생성

1. vercel.com에 GitHub 계정으로 로그인
2. "Add New Project" → NEWNDY-BOARD 저장소 선택
3. **Root Directory를 `apps/web`으로 변경** (핵심!)
   - 모노레포라서 Next.js가 루트가 아니라 `apps/web` 안에 있기 때문
   - Root Directory를 `./`로 두면 빌드 실패
4. Framework Preset: Next.js (자동 감지)
5. Deploy 클릭
